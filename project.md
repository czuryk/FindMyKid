## 1. Описание проекта
Разработка нативного Android-приложения на языке Java для регулярного сбора и отправки гео-координат, а также информации о текущем активном приложении на экране устройства ребенка. Приложение предназначено для родительского контроля. 
Главный приоритет — безопасность, приватность и автономность. Данные должны отправляться исключительно на заданный пользователем сервер (например, на дашборд или другой сервер).

## 2. Основные требования и ограничения
* **Платформа:** Android (Min SDK 26, Target SDK 34+).
* **Язык программирования:** Java.
* **Сторонние библиотеки:** СТРОГО ЗАПРЕЩЕНЫ (за исключением стандартных библиотек AndroidX). 
  * Сетевые запросы: использовать нативный `java.net.HttpURLConnection`.
  * Парсинг/Формирование JSON: использовать нативный `org.json.JSONObject`.
* **Безопасность:** Доступ к настройкам приложения должен быть защищен мастер-паролем. Сохранение настроек производится в `EncryptedSharedPreferences`.

## 3. Функционал приложения

### 3.1. Первичная настройка (First Run)
* При первом запуске приложение запрашивает создание Мастер-пароля.
* Пароль хэшируется (SHA-256) и сохраняется локально в защищенном хранилище.
* Приложение последовательно запрашивает все необходимые системные разрешения (Location, Usage Stats, Battery Optimization).

### 3.2. Экран авторизации
* При каждом последующем открытии приложения (или возврате из фона) отображается экран ввода Мастер-пароля.
* Без ввода правильного пароля доступ к настройкам и остановке сервиса невозможен.
* Восстановить его не возможно, только переустановить приложение.

### 3.3. Экран конфигурации (Main Settings)
Доступен только после ввода Мастер-пароля. Содержит:
1. **Endpoint URL:** Текстовое поле для ввода адреса сервера (например, `https://my-dashboard.local/api/track`).
2. **Частота обновления (Update Interval):** Поле ввода интервала в минутах (например, 5, 10, 15).
3. **Состояние сервиса:** Переключатель (Switch/Toggle) для ручного Запуска/Остановки фонового сервиса слежения.
4. **Кнопка "Изменить пароль":** Смена текущего Мастер-пароля.
5. **Тоглер "отправка координат":** Включено/Выключено.
6. **Тоглер "отправка данных о текущем активном приложении":** Включено/Выключено.
7. **Статус:** Текстовое поле, показывающее время последней успешной отправки данных.

### 3.4. Фоновый сервис (Tracking Service)
* Реализован как `Foreground Service` с постоянным (Ongoing) уведомлением в шторке (требование Android). Текст уведомления: "Синхронизация данных".
* Сервис автоматически стартует при перезагрузке устройства (Boot Receiver).
* По таймеру (согласно настройкам частоты) сервис собирает данные и формирует HTTP POST запрос на указанный Endpoint URL.

### 3.5. Собираемые данные (Payload)
Каждый запрос должен отправлять JSON следующей структуры:
```json
{
  "timestamp": 1710175200,
  "battery_level": 85,
  "location": {
    "latitude": 44.7866,
    "longitude": 20.4489,
    "accuracy_meters": 12.5
  },
  "foreground_app": "com.google.android.youtube"
}
```

## 4. Необходимые разрешения (Permissions)

В `AndroidManifest.xml` должны быть заявлены следующие разрешения:

**Геолокация:**
* `android.permission.ACCESS_COARSE_LOCATION`
* `android.permission.ACCESS_FINE_LOCATION`
* `android.permission.ACCESS_BACKGROUND_LOCATION` (Обязательно для работы в фоне на Android 10+)

**Фоновая работа:**
* `android.permission.FOREGROUND_SERVICE`
* `android.permission.FOREGROUND_SERVICE_LOCATION` (Обязательно для Android 14 / API 34+)
* `android.permission.RECEIVE_BOOT_COMPLETED` (Для автозапуска сервиса после перезагрузки)
* `android.permission.REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` (Для исключения приложения из режима Doze)

**Сеть и Система:**
* `android.permission.INTERNET`
* `android.permission.ACCESS_NETWORK_STATE`
* `android.permission.PACKAGE_USAGE_STATS` (Специальное разрешение для получения имени активного приложения, выдается только через системные настройки)

## 5. Архитектура и реализация (Технические детали)

### 5.1. Модуль геолокации
* Использовать `android.location.LocationManager` (нативный провайдер, `GPS_PROVIDER` и `NETWORK_PROVIDER`).
* Запрашивать координаты непосредственно перед отправкой запроса (Single Update), чтобы не держать GPS-модуль постоянно включенным и экономить батарею.

### 5.2. Модуль определения активного приложения
* Использовать `UsageStatsManager`.
* Логика: Запросить статистику использования за последние 1-2 минуты. Отсортировать по `getLastTimeUsed()`. Приложение с самым свежим timestamp является текущим активным приложением в Foreground.

### 5.3. Модуль сети
* Создать отдельный класс `NetworkSender` (наследующий `Runnable` или использующий `ExecutorService`).
* Использовать `HttpURLConnection`.
* Установить таймауты (`setConnectTimeout`, `setReadTimeout`) в 10-15 секунд.
* Установить заголовок `Content-Type: application/json`.

### 5.4. Модуль хранения настроек
* Использовать библиотеку `androidx.security.crypto.EncryptedSharedPreferences` (официальный пакет AndroidX для безопасного хранения).
* Сохраняемые ключи: `pref_master_password_hash`, `pref_endpoint_url`, `pref_update_interval`, `pref_is_service_enabled`.

### 5.5. Обход ограничений энергосбережения
Приложение должно принудительно перенаправлять пользователя в настройки системы для выдачи разрешений:
1. Игнорировать оптимизацию батареи (`ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`).
2. Доступ к истории использования (`ACTION_USAGE_ACCESS_SETTINGS`).
3. Разрешить локацию "В любом режиме" (Allow all the time).

## 6. Жизненный цикл сервиса
1. Пользователь активирует сервис в UI.
2. Приложение запускает `TrackingService` через `startForegroundService()`.
3. Сервис сразу же вызывает `startForeground()` с привязкой Notification (обязательно указать `foregroundServiceType="location"` в манифесте).
4. Запускается `ScheduledExecutorService` или `Handler` с `Runnable`, который срабатывает каждые N минут.
5. При срабатывании: сбор локации -> сбор активного приложения -> сбор заряда батареи -> POST запрос -> засыпание до следующего тика.
